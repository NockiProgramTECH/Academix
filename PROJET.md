# PROJET.md

## 1. Présentation du projet
**GESTION ECOLE (Academix)** est un projet de gestion scolaire centré sur la **gestion des inscriptions d’élèves**, avec deux interfaces complémentaires :

- **Volet WEB (Django)** : formulaire d’inscription en ligne, dépôt de documents (acte, bulletin, diplôme) + photo optionnelle, API JSON pour l’AJAX, page de confirmation, et administration via Django Admin.
- **Volet DESKTOP (CustomTkinter/Tkinter)** : outil interne (secrétariat) connecté à la **même base SQLite** pour **consulter les inscriptions en attente**, afficher la photo, **valider** le dossier, et (dans un script séparé) effectuer une **répartition/affectation officielle** des élèves dans des divisions (ex: “6EME A, 6EME B…”).

Le projet stocke les fichiers uploadés dans `WEB/media/` et s’appuie sur `WEB/db.sqlite3` comme base de données principale.

## 5. Logique générale du fonctionnement

Le projet suit une logique “pipeline” :

1. **Inscription WEB** : l’élève (ou utilisateur) remplit le formulaire et envoie les pièces.
2. **Persistance** : Django crée les enregistrements en base (`Eleve`, `DocumentEleve`, éventuellement `Tuteur`) et stocke les fichiers sous `WEB/media/`.
3. **Statut** : l’élève est créé avec un statut de type “en attente” (champ `statut` dans la table `Inscriptions_eleve`).
4. **Traitement interne** : l’application **DESKTOP** se connecte à la même base SQLite et affiche les dossiers en attente.
5. **Validation** : un opérateur accepte/valide le dossier (mise à jour du statut).
6. **Scolarité / Affectation** : un script desktop peut répartir les élèves validés dans des divisions et synchroniser les tables “officielles” (`Classes`, `Scolarite_Affectation`) + renseigner `classe_reelle`.

## 6. Étapes logiques du projet (workflow)

### Workflow “Inscription”
- **GET** page d’inscription : `/inscriptions/`
- **POST** inscription (AJAX) : `/inscriptions/api/inscription/`
  - validation des champs
  - validation des documents (PDF/DOCX, taille max 5MB)
  - validation photo (JPG/PNG/WEBP, taille max 2MB)
  - création `Eleve` (statut “EN_ATTENTE”)
  - création `DocumentEleve`
  - création `Tuteur` si informations présentes
  - réponse JSON (incluant `redirect_url`)
- **GET** confirmation : `/inscriptions/confirmation/<uuid>/`

### Workflow “Validation (desktop)”
- Chargement de la liste : requête SQL sur `Inscriptions_eleve` filtrée sur `statut = 'EN_ATTENTE'`
- Sélection d’un élève → affichage détails + photo
- Action “ACCEPTER” → update du champ `statut`

### Workflow “Affectation officielle (script)”
- Sélection des élèves validés d’un niveau (ex: 6EME)
- Création (si nécessaire) des libellés de classes “6EME A/B/C…”
- Insertion des lignes d’affectation dans `Scolarite_Affectation`
- Mise à jour de `classe_reelle` sur `Inscriptions_eleve`

## 9. Points à corriger ou améliorer (sans modifier le code)

### Cohérence “statut” entre Web et Desktop
- Le modèle Django `Eleve.statut` est défini par choix (`EN_ATTENTE`, `VALIDE`, `REJETE`), mais côté desktop/scripts on voit des valeurs telles que `ACCEPTE` / `ACCEPTED`.  
  **Risque** : incohérences de données et filtres qui ne matchent plus.

### Gestion des dossiers/media : incohérences code vs documentation
- La documentation `WEB/GUIDE_DEMARRAGE_RAPIDE.md` / `DIAGRAMMES_FLUX.md` décrit une structure `media/inscriptions/{classe}/photos/{nom}/…`, alors que :
  - `models.get_image()` et `document_upload_path()` utilisent plutôt `inscriptions/{classe}/{uuid}/...`
  - `Inscriptions/utils.py` mélange les paramètres (`id`, `nom_complet`) pour construire les dossiers
  - `delete_eleve_folder()` référence une variable `id` non définie (risque d’échec à l’exécution)
  **Risque** : la commande `rebuild_student_folders` et les tests supposent une structure qui ne correspond pas forcément aux chemins réellement utilisés.

### Django “management command” et packaging Python
- Le dépôt ne contient pas de `__init__.py` (aucun n’est présent dans l’arborescence), ce qui peut impacter la reconnaissance de certains sous-paquets selon contexte.  
  **À vérifier** : découverte de la commande `rebuild_student_folders` par Django.

### Templates / routes
- `Inscriptions/views.py` tente de rendre `inscriptions/liste_inscriptions.html` mais le template est absent.
- `base.html` pointe “Inscription” sur `/` alors que l’URL d’inscription est `/inscriptions/`.  
  **Risque** : page d’accueil non servie à l’URL racine.

### Sécurité / configuration
- `SECRET_KEY` est committée, `DEBUG=True`, `ALLOWED_HOSTS=["*"]`.  
  **Risque** : configuration non prête pour production.
- `MEDIA_URL` vaut `media/` (sans slash initial).  
  **Risque** : liens media inattendus selon configuration et templates.

### Qualité / dette technique
- Fichiers binaires (ex: `WEB/db.sqlite3`, `WEB/media/*`) présents dans le dépôt.  
  **Risque** : repo lourd, collisions de données, difficulté de versionning.
- Code “prototype / test” mélangé au code produit (`WEB/test.py`, `WEB/p.py`, `testfiz.py`, `DESKTOP/testkinter.py`).

## 12. Conclusion technique

Le projet est une base fonctionnelle orientée **inscription** avec une architecture “double interface” :

- **Django** pour la saisie et le stockage (DB + media).
- **Desktop** pour le traitement interne et la scolarité (validation + affectation).

Les fonctionnalités cœur (inscription + stockage + validation côté desktop) sont présentes, mais l’état actuel montre des **incohérences importantes** (chemins media/dossiers, valeurs de statut, templates manquants, intégration partielle de `AppNote`) et une **configuration non durcie** (mode debug, secret en clair, DB/media versionnés). Une consolidation (standardisation des conventions et séparation code/données) rendrait l’ensemble bien plus robuste et maintenable.
